# CLAUDE.md — cosmos-desktop 작업 지침

> Claude Code가 이 프로젝트에서 반드시 지켜야 하는 규칙.
> 매 세션 시작 시 먼저 읽고, 의심스러운 순간마다 다시 참조한다.

---

## 0. 프로젝트 개요

**cosmos-desktop** 은 기존 파일 탐색기를 대체하는, 무한 줌 가능한 2D/3D 우주 캔버스 데스크톱 앱이다.

- 파일/링크/메모를 무한 3D 공간에 자유 배치
- 프로젝트별 "행성(클러스터)"로 관련 파일을 근처에 모음
- 줌아웃 시 별자리처럼, 줌인 시 개별 파일 조작
- 로컬 파일 시스템 경로 ↔ 시각적 좌표 매핑

**성격:** 실사용 + 포트폴리오. "내가 매일 쓰고 싶은 수준"이 품질 기준. 프로토타입 타협 금지.

**기술적 핵심 난제:** 대용량 노드 렌더링 성능, 무한 캔버스 상태 관리, 로컬 FS 동기화

## 2. 최종 기술 스택 (확정)

| 레이어 | 선택 | 비고 |
|---|---|---|
| 데스크톱 프레임워크 | **Tauri 2.x** | Rust 백엔드, 번들 크기 작음 |
| 렌더러 | **three.js (순수)** | React-three-fiber 사용 금지 |
| 상태 관리 | **Zustand + TypedArray 이중 구조** | 고빈도는 TypedArray, 저빈도만 Zustand |
| 파일 감시 | **notify (Rust crate)** | 재귀 감시 + rescan fallback |
| 지속성 | **SQLite (tauri-plugin-sql)** | 좌표, 메타데이터, 사용자 설정 |
| 공간 인덱스 | **rbush** | 히트테스트, 뷰포트 컬링 |
| 텍스트 렌더 | **MSDF bitmap font** | 노드당 Text 객체 금지 |
| 언어 | **TypeScript + Rust** | |
| 플랫폼 | Windows / macOS (Linux는 나중) | |

**3D/2D 모드:** 처음부터 three.js로 3D 아키텍처. 초기 MVP는 **카메라 잠금(orthographic top-down)** 으로 2D처럼 동작. 나중에 카메라 토글만 풀어 3D로 확장 (아키텍처 재작성 아님).

근거와 의사결정 로그는 `docs/ARCHITECTURE.md` 참조.

## 3. 컨텍스트 관리

- **이 파일은 300줄 이내로 유지.** 상세 문서는 `docs/` 하위에 Lazy Loading.
- 문서 구조:
  - `docs/ARCHITECTURE.md` — 아키텍처 상세 + 의사결정 로그
  - `docs/RENDERING.md` — three.js / WebGL 전략 (필요 시 생성)
  - `docs/FS_SYNC.md` — notify + SQLite 동기화 (필요 시 생성)
  - `docs/STATE.md` — 이중 상태 아키텍처 상세 (필요 시 생성)
- **세션당 하나의 기능에만 집중.** 70% 차면 즉시 `/clear` 또는 `/compact`.
- 대량 데이터(로그, 벤치마크, FS 스캔)는 **스크립트로 처리**하고 결과값만 컨텍스트로.

## 4. 작업 실행 프로세스

### 4-1. Plan 모드 우선
- 코드 수정/생성 전 **반드시 Plan 제시 → 사용자 승인**.
- Plan에는: 수정/생성 파일 목록, 각 파일에서 할 일, 예상 위험, 테스트 전략.
- "go", "진행해" 같은 명시적 승인 전에 파일 건드리지 않는다.

### 4-2. 설계 결정은 Gemini + Planner와 상의
- 중요한 아키텍처/스택/트레이드오프 결정은 혼자 하지 않는다.
- 기본 흐름: Gemini(기획 의견) → Planner(비판적 검증) → Tech Lead(사용자에 요약 보고) → 사용자 결정.
- **Gemini 호출은 기본 Flash 모델** (Pro 쿼터 아끼려고). Deep Research가 정말 필요할 때만 Pro.

### 4-3. TDD
- 순수 로직(좌표 변환, 상태 리듀서, FS 경로 매핑, SQL 빌더 등)은 **테스트 먼저, 구현 다음**.
- 렌더링/UI 시각 요소는 스냅샷 또는 수동 검증 허용.
- Rust 로직은 `#[cfg(test)]` 단위 테스트 기본값.

## 5. 에이전트 호출 가이드라인

- 단일 레이어의 작은 수정

### 5-2. 언제 에이전트 호출

- **Frontend:** three.js/React/Zustand/셰이더 작업이 5파일 이상 또는 새 렌더링 기능
- **Backend:** Rust/Tauri/notify/SQLite 작업이 새 모듈 단위이거나 IPC 신설
- **QA:** 기능 구현 완료 후 테스트 보강, 또는 버그 재현 테스트 작성
- **병렬 팀 소집:** 20+ 파일 교차 수정, 여러 레이어 동시 변경

### 5-3. 작업 흐름
1. **Plan** — Tech Lead가 작업을 쪼갠다
2. **분배** — 독립 작업은 병렬, 종속 작업은 순차

4. **QA** — qa 에이전트가 테스트 / 엣지 케이스 검증

## 6. 코드 작성 규칙

### 6-1. 한국어 주석 필수
사용자는 코드를 **읽고 이해하며 배우는** 입장이다. 코드는 읽을 수 있어야 한다.

- 함수 단위: 무엇을, 왜 하는지 3~5줄 한국어 주석
- **Rust 개념 처음 등장:** "이건 Rust의 ___ 개념" 간단 설명 (소유권, `Result<T,E>`, `Arc<Mutex<T>>`, `?`, 매크로 등)
- **3D/WebGL 수학 등장:** "이 행렬은 카메라를 ___하는 것" 직관적 풀이
- **성능 최적화 트릭:** "이걸 이렇게 한 이유는 ___"
- 복잡한 로직은 줄 단위 주석

### 6-2. 품질 기준
- 60fps @ 10,000 노드 / 최소 30fps @ 100,000 노드
- 초기 메모리 < 150MB, 노드 10k 로드 < 500MB
- 1시간 사용 메모리 누수 없음
- 한글/이모지 파일명 정상 동작
- 에러는 숨기지 말고 한국어 사용자 메시지로 변환

### 6-3. 에러 처리
- Rust: `unwrap()` 프로덕션 금지 (테스트 OK), `?`로 전파
- TS: 에러를 삼키지 말고 UI 레이어까지 올려서 사용자에게 보여줌
- 모든 에러 로그는 "왜 났는지" 맥락 포함

## 8. 커뮤니케이션 규칙

- **한국어로 응답.** 기술 용어는 영어 유지 가능.
- 커밋 메시지는 한국어 우선 (Conventional Commits: `feat: ...`, `fix: ...`).
- 불확실한 것은 "확실하지 않음" 명시.
- 에러 숨기지 말고 그대로 보고 + 원인 분석.
- 긴 작업은 중간 경과 간단히 보고.
- 추측보다 질문. 결정은 사용자가 한다.

### 8-1. Bash tool 주의사항 (Windows PATH 캐시 이슈)

> 이 프로젝트는 Windows + Git Bash 환경에서 돌아간다. Bash에는 명령어
> 경로를 해시로 캐싱해두는 동작이 있어서, 세션 중에 PATH가 바뀌거나
> 새 바이너리가 설치되면 Claude가 "command not found"로 잘못 판단하는
> 경우가 발생한다. 아래 규칙을 항상 지킨다.

- **캐시된 "없음"을 믿지 않는다.** `cargo`, `npm`, `tauri`, `rustc` 등
  필수 툴이 "not found"로 나오면, 먼저 `hash -r` 로 bash 해시 테이블을
  리셋하고 다시 시도한다. 그래도 안 되면 절대 경로 호출 시도
  (`/c/Users/User/.cargo/bin/cargo --version` 등).
- **설치 직후 재확인.** `npm create tauri-app`, `cargo install`,
  `rustup` 같은 명령으로 PATH가 갱신된 직후에는 `hash -r` 후 재검증.
  새 shell이 아니면 PATH 변경이 반영되지 않을 수 있다.
- **"없다"고 단정하기 전에 3단계 확인.** (1) `hash -r` 후 재시도,
  (2) `which <cmd>` 로 실제 경로 조회, (3) 절대 경로로 직접 호출.
  셋 다 실패해야 진짜로 "설치 안 됨"이다.
- **사용자에게 설치 요청은 신중히.** 위 3단계를 거치지 않은 상태에서
  "설치해 주세요"라고 말하면 사용자는 이미 설치된 툴을 다시 설치하게
  된다. 매우 비효율적이므로 금지.
- **경로 구분자.** bash 환경이므로 `/` 사용, `NUL` 대신 `/dev/null`.
  Windows 경로를 인자로 넘길 땐 `/c/Users/...` 또는 따옴표 감싼
  `"C:\Users\..."`.

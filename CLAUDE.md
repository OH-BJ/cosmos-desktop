# CLAUDE.md — cosmos-desktop 프로젝트 작업 지침

> 이 문서는 Claude Code가 이 프로젝트에서 작업할 때 **반드시 지켜야 하는 규칙**입니다.
> 매 세션 시작 시 이 문서를 먼저 읽고, 의심스러운 순간마다 다시 참조하세요.

---

## 0. 프로젝트 개요

**cosmos-desktop** 은 기존 파일 탐색기를 대체하는, 무한히 줌인/줌아웃되는 2D/3D 우주 캔버스 데스크톱 앱입니다.

- 파일/링크/메모를 무한 공간에 자유 배치
- 프로젝트별로 "행성(클러스터)"을 만들어 관련 파일들을 근처에 모음
- 줌아웃 시 별자리처럼 보이고, 줌인 시 개별 파일 조작 가능
- 로컬 파일 시스템 경로 ↔ 시각적 좌표 매핑

**기술적 핵심 난제:** 대용량 노드 렌더링 성능, 무한 캔버스 상태 관리, 로컬 FS 동기화

**협업 구조:** Gemini(MCP 연결됨) = 기획/아키텍처/추론 담당, Claude Code = 구현/리뷰/리팩터링 담당

## 1. 컨텍스트 관리 (가장 중요)

### 1-1. Lazy Loading 원칙
- **이 `CLAUDE.md` 파일은 300줄 이내로 유지한다.**
- API 스펙, DB 스키마, 상세 아키텍처 문서 등 방대한 자료는 별도 `.md` 파일로 분리하고, 필요할 때만 참조한다.
- 예시 구조:
  - `docs/ARCHITECTURE.md` — 전체 아키텍처 상세
  - `docs/RENDERING.md` — WebGL/Canvas 렌더링 전략
  - `docs/FS_SYNC.md` — 파일시스템 동기화 로직
  - `docs/STATE.md` — 상태 관리 설계

### 1-2. 세션당 하나의 기능
- **한 세션에서는 하나의 기능 구현에만 집중한다.**
- 예: "오늘은 무한 캔버스 렌더링만", "오늘은 FS watcher만"
- 기능이 끝나거나 컨텍스트가 70% 이상 차오르면 **즉시 `/clear` 또는 `/compact`** 를 사용해 컨텍스트를 비운다. 환각 방지 목적이다.

### 1-3. 무거운 작업은 스크립트로 분리
- 대량 데이터 처리(로그 파싱, 파일 대량 스캔, 벤치마크 결과 집계 등)는 Claude가 직접 파싱하지 않는다.
- 대신 **Python 또는 Bash 스크립트를 작성해 실행하고, 최종 결과값만 컨텍스트로 가져온다.**

## 2. 작업 실행 프로세스

### 2-1. Plan 모드 우선
- **코드를 수정하거나 새 기능을 추가하기 전, 반드시 Plan 모드로 계획을 먼저 보고하고 사용자 승인을 받는다.**
- Plan에는 다음이 포함되어야 한다:
  - 수정/생성할 파일 목록
  - 각 파일에서 할 일
  - 예상 위험 요소
  - 테스트 전략
- 사용자가 "go", "진행해" 같은 명시적 승인을 하기 전에는 절대 파일을 수정하지 않는다.

### 2-2. 설계 결정은 Gemini와 상의
- 중요한 아키텍처/기술 스택/트레이드오프 결정은 혼자 하지 않는다.
- **MCP로 연결된 Gemini에게 먼저 의견을 구한 뒤**, 그 분석을 사용자에게 요약 보고하고 결정을 받는다.
- 예: "Electron vs Tauri", "상태관리에 Zustand vs Jotai vs Valtio", "렌더링에 PixiJS vs three.js vs 순수 WebGL"

### 2-3. TDD 기반 코딩
- 코드를 수정하거나 새 기능을 추가할 때, **관련 단위 테스트를 먼저 작성하거나 갱신하고**, 통과를 확인한 후 다음 단계로 넘어간다.
- 순수 로직(좌표 변환, 상태 리듀서, FS 경로 매핑 등)은 특히 TDD를 엄격히 적용한다.
- 렌더링/UI 같은 시각적 부분은 스냅샷 테스트 또는 수동 검증으로 대체 가능하다.

## 3. 에이전트 & 병렬 작업

### 3-1. 단순 작업은 단일 에이전트로
- 이 프로젝트는 초기 단계이므로, **기본은 단일 에이전트로 진행한다.**
- 서브 에이전트 팀은 토큰을 많이 먹으므로 초기에는 과도하다.

### 3-2. 서브 에이전트 병렬 실행 조건
- **20개 이상의 파일을 동시에 다뤄야 하는 복잡한 작업일 때만** 서브 에이전트 팀을 구성한다.
- 구성 원칙:
  - 독립적인 작업(프론트엔드 / 백엔드 / FS 레이어)은 병렬로 분리
  - **QA 담당 에이전트는 Haiku 모델**로 설정해 토큰 낭비 방지
  - 테스트 환경 세팅은 백엔드 에이전트가 미리 완료 후 프론트 에이전트에게 넘김

### 3-3. 반복 업무는 Skills로 모듈화
- 코드 리뷰, 보고서 작성, 커밋 메시지 작성 등 **반복되는 작업은 `.claude/skills/` 폴더 내에 마크다운 파일로 모듈화**하고 온디맨드로 호출한다.

## 4. 자동화 (Hooks)

### 4-1. Hooks 활용
- 다음은 Hook으로 자동화한다:
  - 코드 작성 완료 후 자동 lint + format (예: Prettier, ESLint)
  - 세션/작업 종료 시 알림
  - 커밋 전 테스트 자동 실행
- Hook 설정은 `.claude/hooks/` 또는 프로젝트의 `settings.json`에 정의한다.

## 5. 보안

### 5-1. 비밀 정보 관리
- **API 키, 토큰, 개인정보는 절대 코드에 하드코딩하지 않는다.**
- 모든 민감 정보는 `.env` 파일로 분리하고, `.env`는 반드시 `.gitignore`에 포함시킨다.
- `.env.example` 파일을 커밋해 필요한 환경변수 키 이름만 공유한다.
- 로그/에러 메시지에도 비밀 정보가 노출되지 않도록 확인한다.

### 5-2. 로컬 파일 접근
- 이 프로젝트는 로컬 파일 시스템에 광범위하게 접근한다. 다음을 지킨다:
  - 사용자가 지정한 디렉터리 **외부**는 절대 접근/수정하지 않는다.
  - 파일 삭제는 반드시 **사용자의 명시적 확인** 후에만 실행한다.
  - 시스템 디렉터리(`C:\Windows`, `C:\Program Files` 등)는 접근 금지.

## 6. 커뮤니케이션 규칙

- **한국어로 응답한다.** 코드 주석/커밋 메시지도 한국어 우선 (기술 용어는 영어 유지 가능).
- 불확실한 것은 추측하지 말고 **"확실하지 않음"을 명시**한다.
- 에러가 나면 숨기지 말고 그대로 보고한다.
- 긴 작업은 중간 경과를 간단히 보고한다.

## 7. 세션 시작 체크리스트

새 세션을 시작할 때 Claude Code는 다음을 순서대로 한다:

1. 이 `CLAUDE.md` 파일을 읽었음을 확인한다.
2. `/mcp` 로 Gemini 연결 상태를 확인한다.
3. 오늘의 목표 기능 **하나**를 사용자와 확인한다.
4. 관련 문서(`docs/` 하위)가 있다면 필요한 것만 읽는다.
5. Plan 모드로 계획을 먼저 제시한다.
6. 사용자 승인을 받은 뒤 구현을 시작한다.

## 9. 토큰 최적화 규칙 (Token Optimization)

이 규칙은 불필요한 토큰 낭비를 막고 작업 품질을 높이기 위한 절대 규칙이다.
이전 규칙(§1~§8)과 충돌 시 이쪽이 우선.

### 9-1. 파일 탐색 및 읽기
- **중복 읽기 금지**: 이미 한 번 읽고 분석한 파일은 다시 읽지 않는다. 내용이 기억나지 않거나 수정된 경우에만 부분적으로 다시 읽는다 (`view_range` 활용).
- **범위 제한**: 거대한 파일을 통째로 컨텍스트에 올리지 마라. 수정이 필요한 '특정 함수/특정 라인'만 타겟팅.
- **무시 디렉터리**: `node_modules`, `target`, `dist`, `.next`, `build` 폴더는 절대 탐색하지 않는다.
- **이미 컨텍스트에 있는 파일은 다시 읽지 않는다**. 대화 초반에 봤다면 그대로 활용.

### 9-2. 작업 및 실행
- **Plan 모드 우선**: 코드를 대량 수정하거나 새 기능을 추가하기 전, 반드시 Plan 보고 → 사용자 승인 → 구현.
- **Diff-only 수정**: `str_replace` 사용. 파일 전체 재작성 금지.
- **간결성 원칙**: 인사말, 자명한 설명, 작업 메타 코멘트 ("이제 ~를 하겠습니다") 생략. 그러나 "왜 이렇게 했나" 한두 줄 학습 목적 주석은 유지 (사용자가 학습 중).
- **자기 보고 압축**: 단계 완료 시 5줄 이내 요약. 표로 정리할 때만 확장.

### 9-3. 도구 및 쉘 명령어
- **병렬 실행**: 독립적 도구 호출은 반드시 병렬. 순차 호출 금지.
- **출력 필터링**: 방대한 출력 명령어 금지.
  - `git log` → `git log --oneline -5` 또는 `head -10`
  - `git status` → `git status --short`
  - 테스트 출력 → 실패 케이스만 `grep -E "FAIL|Error"`
  - `find` 결과 → `head` 또는 `wc -l`
  - 디렉터리 트리 → `tree -L 2` 또는 `ls -la` 한 단계만
- **로그 파일 → 마지막 N줄만**: `tail -50` 기본.

### 9-4. 모델 및 서브에이전트 운영
- **Extended Thinking 활용 기준**:
  - 사용 OK: 아키텍처 결정, 다중 AI 리뷰 통합, 디버깅 원인 분석, M2 같은 큰 마일스톤 Plan
  - 사용 금지: 오타 수정, import 정리, 단순 파일 검색, 이미 결정된 사항 재확인
- **서브에이전트 모델 배정**:
  - **QA**: Haiku (단순 테스트 실행/검증)
  - **Frontend / Backend**: Sonnet (코드 작성/리뷰 품질 필요)
  - **Planner**: Sonnet 또는 Opus (메타리뷰/비판적 검토 필요)
- **서브에이전트 호출 기준**:
  - 호출 OK: 멀티 파일 작업, 도메인 전문성 필요 작업, 교차 리뷰
  - 호출 금지: 한 파일 수정, 단순 검색, Tech Lead 단독으로 충분한 작업
  - 호출 시 명시: "왜 이 에이전트가 필요한가" 한 줄 근거.

### 9-5. 세션 관리
- **체크포인트 설계**: 긴 작업은 논리적 체크포인트(커밋 직전, Plan 단계 완료 등)에서 자연 분할. 토큰 80% 도달 시 사용자에게 알림.
- **컨텍스트 리셋 권장 시점**: 마일스톤 완료 후, 또는 주제가 바뀔 때 `/clear` 권장. Plan은 `docs/PLAN.md`에 저장되어 있으므로 안전.
- **서브에이전트 호출은 컨텍스트 10배 소모 인식**: 각 에이전트는 독립 컨텍스트로 시작하므로 토큰 비용이 큼. 단순 작업은 Tech Lead 단독.

### 9-6. 중요 예외 (이 규칙보다 우선)
- 사용자 안전 (보안 검증, 위험한 명령어 확인)
- 데이터 무결성 (커밋 전 git diff 확인)
- 사용자가 명시적으로 "자세히" 요청한 경우
